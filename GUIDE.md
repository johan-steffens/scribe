# Scribe User Guide

This guide walks through every feature of Scribe in depth. It assumes you have
Scribe installed (see the
[README](README.md#installation) for binary download and source build
instructions) and a terminal open.

---

## Introduction

Scribe has two modes:

| Mode | How to launch | Best for |
|------|---------------|----------|
| **CLI** | `scribe <subcommand>` | Scripting, quick one-liners, piping to `jq` |
| **TUI** | `scribe` (no arguments) | Interactive planning sessions, navigation |

Both modes share the same database. A task created via the CLI appears
immediately in the TUI the next time it refreshes, and vice versa.

Everything works fully offline by default. Optional sync lets you mirror your
state across multiple machines — see the [Sync](#sync) section.

---

## Core concepts

### Projects

A project is a container for related work. Every task, todo, time entry, and
reminder must belong to a project. Projects have a status (`active`, `paused`,
`completed`) and can be archived.

### The `quick-capture` project

A project with `slug = "quick-capture"` is created automatically on first run.
It is the default destination for any item created without an explicit
`--project` flag. It cannot be archived or deleted (`is_reserved = 1`). Use it
as a staging area and move items to real projects as you go.

### Slugs

Every entity has a **globally unique slug** — a human-readable string used
everywhere in the CLI instead of numeric IDs.

| Entity | Slug format | Example |
|--------|-------------|---------|
| Project | User-defined kebab-case | `payments` |
| Task | `{project}-task-{title-slug}` | `payments-task-fix-login` |
| Todo | `{project}-todo-{title-slug}` | `payments-todo-review-docs` |
| Reminder | `{project}-reminder-{message-slug}` | `payments-reminder-deploy-friday` |
| TimeEntry | `{project}-entry-{YYYYMMDD}-{HHmmss}` | `payments-entry-20260331-143000` |
| CaptureItem | `capture-{YYYYMMDD}-{HHmmss}` | `capture-20260331-143000` |

Slugs are auto-generated by lowercasing the title, replacing non-alphanumeric
characters with `-`, and truncating. A 4-character random suffix is appended
only when a collision is detected. Project slugs are user-defined and must be
valid kebab-case.

### Archiving vs. deleting

There are no immediate hard deletes. Every entity has an `archived_at`
timestamp. Archiving sets this timestamp; restoring clears it. Archived items
are hidden from default list views.

To permanently delete an item you must archive it first, then delete it:

```sh
scribe task archive payments-task-fix-login
scribe task delete payments-task-fix-login
```

When a project is archived, all its linked tasks, todos, time entries, and
reminders are archived in the same operation. Restoring the project does
**not** automatically restore those items — you restore them individually.

### The inbox

`CaptureItem`s are raw, unstructured thoughts stored in the inbox table. They
have no project binding by default. Use `scribe capture` to add them and
`scribe inbox process` to categorise them later.

## Projects

### Creating a project

```sh
scribe project add payments --name "Payments Integration"
scribe project add payments --name "Payments Integration" --desc "Stripe and PayPal flows"
```

The first positional argument is the slug. It must be unique across all
projects.

### Listing projects

```sh
# Active projects (default)
scribe project list

# Only completed projects
scribe project list --status completed

# Include archived projects
scribe project list --archived
```

### Showing a project

```sh
scribe project show payments
```

Output (text):

```
slug:        payments
name:        Payments Integration
status:      active
description: Stripe and PayPal flows
created:     2026-03-31 14:00 UTC
updated:     2026-03-31 14:00 UTC
```

### Editing a project

Any combination of flags may be used; omitted fields are unchanged.

```sh
# Rename the display name
scribe project edit payments --name "Payments v2"

# Change status
scribe project edit payments --status paused

# Rename the slug itself
scribe project edit payments --new-slug payments-v2

# After renaming, the old slug no longer exists; use the new one
scribe project show payments-v2
```

### Archiving and restoring

```sh
scribe project archive payments
# All linked tasks, todos, time entries, and reminders are also archived.

scribe project restore payments
# The project is active again, but its items remain archived.
```

Reserved projects (`quick-capture`) cannot be archived:

```
error: project 'quick-capture' is reserved and cannot be archived
```

### Deleting

A project must have no linked items before it can be deleted. Archive it first
(which cascades to all items), then delete:

```sh
scribe project archive payments
scribe project delete payments
```

If the project still has active items the delete will fail with a database
constraint error.

## Tasks

### Creating a task

```sh
# Defaults to quick-capture project, medium priority, no due date
scribe task add "Fix the login bug"

# With all options
scribe task add "Fix the login bug" \
  --project payments \
  --priority high \
  --due 2026-04-15
```

The slug is auto-generated from the project slug and the title:

```
Created task: Fix the login bug (payments-task-fix-login)
```

### Listing tasks

```sh
# All active tasks
scribe task list

# Filter by project
scribe task list --project payments

# Filter by status
scribe task list --status in_progress

# Filter by priority
scribe task list --priority urgent

# Include archived
scribe task list --archived
```

Filters can be combined: `--project payments --status todo --priority high`.

### Showing a task

```sh
scribe task show payments-task-fix-login
```

### Editing a task

```sh
scribe task edit payments-task-fix-login --status in_progress
scribe task edit payments-task-fix-login --priority urgent --due 2026-04-10
scribe task edit payments-task-fix-login --title "Fix the login redirect bug"
```

### Marking done

```sh
scribe task done payments-task-fix-login
```

This sets `status = "done"`. Equivalent to `--status done` via edit.

### Moving between projects

```sh
scribe task move payments-task-fix-login --project quick-capture
```

The task slug retains the original project prefix even after moving; the slug
itself does not change.

### Archiving and restoring

```sh
scribe task archive payments-task-fix-login
scribe task restore payments-task-fix-login
```

## Todos

Todos are simpler than tasks. They have a title, a done flag, and a project —
no priority, no status field beyond done/undone, no due date. They suit
checklists and one-off items.

### Creating a todo

```sh
# Defaults to quick-capture
scribe todo add "Review PR #42"

# Under a specific project
scribe todo add "Review PR #42" --project payments
```

Slug format: `payments-todo-review-pr-42`.

### Listing todos

```sh
# Only undone todos (default)
scribe todo list

# Include done todos
scribe todo list --all

# Filter by project
scribe todo list --project payments --all

# Show archived
scribe todo list --archived
```

### Marking done

```sh
scribe todo done payments-todo-review-pr-42
```

There is no `undone` CLI command; toggle done/undone in the TUI with `Space`,
or re-create the todo.

### Moving between projects

```sh
scribe todo move payments-todo-review-pr-42 --project api
```

### Archiving, restoring, deleting

```sh
scribe todo archive payments-todo-review-pr-42
scribe todo restore payments-todo-review-pr-42

# Must be archived first
scribe todo archive payments-todo-review-pr-42
scribe todo delete payments-todo-review-pr-42
```

### Showing a todo

```sh
scribe todo show payments-todo-review-pr-42
```

## Time Tracking

### Starting a timer

Only one timer can run at a time.

```sh
# Minimal — uses quick-capture project
scribe track start

# Under a project
scribe track start --project payments

# Linked to a specific task
scribe track start --task payments-task-fix-login

# With a free-text note
scribe track start --project payments --note "Debugging auth middleware"
```

A time entry slug is generated automatically:

```
Timer started: payments-entry-20260331-143000
```

### What happens if a timer is already running

```sh
scribe track start --project payments
# error: timer 'payments-entry-20260331-143000' is already running (5m 30s elapsed);
#        stop it first with `scribe track stop`
```

### Stopping the timer

```sh
scribe track stop
```

If no timer is running:

```
error: no timer is currently running
```

### Checking status

```sh
scribe track status
```

Returns the active entry slug, linked task (if any), and elapsed duration. If
no timer is running, indicates idle.

### Reports

```sh
# Today's completed entries
scribe track report --today

# This week's completed entries
scribe track report --week

# Filter by project
scribe track report --project payments

# Combine
scribe track report --week --project payments
```

Reports include only completed (stopped), non-archived entries.

### Archiving a time entry

Time entries are archived through the TUI. Press `D` in the Tracker view to
archive the selected entry (confirmation required). There is no CLI subcommand
for archiving individual time entries.

## Quick Capture and Inbox

### Capturing a thought

```sh
scribe capture "Look into Redis caching for the API"
# Captured: Look into Redis caching for the API [capture-20260331-143512]
```

No project or context is required. The slug is timestamp-based.

### Viewing the inbox

```sh
# Unprocessed items only (default)
scribe inbox list

# All items including already-processed ones
scribe inbox list --all
```

### Processing an inbox item

```sh
scribe inbox process capture-20260331-143512
```

The CLI enters interactive mode:

```
Item: Look into Redis caching for the API
  [1] Convert to task
  [2] Convert to todo
  [3] Assign to project
  [4] Discard
Choice:
```

**Option 1 — Convert to task**: prompts for a project slug and an optional
title. If you leave the title blank, the capture body is used verbatim.

**Option 2 — Convert to todo**: prompts for a project slug and an optional
title.

**Option 3 — Assign to project**: validates the project exists and marks the
item processed. No task or todo is created; it is a logical association only.

**Option 4 — Discard**: marks the item processed without creating anything.

In all cases the capture item is marked `processed = true` and no longer
appears in `scribe inbox list`.

### Workflow

The intended pattern is:

1. During the day, dump thoughts with `scribe capture "..."` without switching
   context.
2. At a natural break, run `scribe inbox list` to see what accumulated.
3. Work through items with `scribe inbox process <slug>`.

## Reminders

### Creating a reminder

`--project` and `--at` are required.

```sh
# Absolute datetime
scribe reminder add \
  --project payments \
  --at "2026-04-01 14:00" \
  --message "Deploy the Stripe integration"

# Link to a task
scribe reminder add \
  --project payments \
  --task payments-task-fix-login \
  --at "2026-04-01 09:00" \
  --message "Check if login fix is deployed"
```

### Supported datetime formats for `--at`

| Input | Interpretation |
|-------|----------------|
| `2026-04-01T14:00:00` | ISO 8601 with T separator, local time |
| `2026-04-01 14:00` | Space-separated, local time |
| `2026-04-01` | Date only, midnight local time |
| `tomorrow 09:00` | Next calendar day at the given time |
| `friday 17:00` | Coming Friday at 17:00 |
| `friday` | Coming Friday at 09:00 |

All local times are converted to UTC for storage. "Coming" weekday means at
least 1 day ahead; if today is Friday, `friday` resolves to next Friday.

### Listing reminders

```sh
scribe reminder list
scribe reminder list --project payments
scribe reminder list --archived
```

### Showing a reminder

```sh
scribe reminder show payments-reminder-deploy-the-stripe-integration
```

### How firing works

Reminders are delivered as OS desktop notifications in three situations:

1. **On any `scribe` invocation** — at startup Scribe runs `check_due`, which
   queries for all reminders with `remind_at <= now` and `fired = false`,
   marks them `fired = true`, and sends a desktop notification for each one.

2. **While the TUI is open** — a background thread polls `check_due` every
   30 seconds and fires notifications in real time without requiring a restart.

3. **Via `scribe daemon`** — a long-running background process that polls
   every 30 seconds continuously, even when the TUI is not open.

On **macOS** notifications are delivered via `osascript` and appear attributed
to **Script Editor** in Notification Centre. To ensure they are visible, go to
**System Settings → Notifications → Script Editor** and set the alert style to
Banners or Alerts.

On **Linux** notifications are delivered over D-Bus (requires a notification
daemon such as `dunst`, which ships with most desktop environments).

On **Windows** notifications are delivered via WinRT.

Fired reminders remain visible in `scribe reminder list` until archived.

### Archiving and deleting

```sh
scribe reminder archive payments-reminder-deploy-the-stripe-integration
scribe reminder restore payments-reminder-deploy-the-stripe-integration

# Must be archived first
scribe reminder archive payments-reminder-deploy-the-stripe-integration
scribe reminder delete payments-reminder-deploy-the-stripe-integration
```

### Running the background daemon

For automatic notification delivery, install the daemon as a system service
using `scribe setup` (recommended) or `scribe service install`:

```sh
# First-run wizard — offers daemon service and agent integration
scribe setup

# Or install the daemon service directly
scribe service install

# Check current service status
scribe service status

# Remove the service
scribe service uninstall
```

`scribe service install` writes a launchd user agent (macOS) or systemd user
unit (Linux), loads it immediately, and records the installation in
`~/.config/scribe/config.toml` so `scribe setup` can report it. No `sudo` is
required on either platform.

You can also run the daemon manually in a terminal (useful for testing):

```sh
# Default 30-second polling interval
scribe service run

# Shorter interval for testing
scribe service run --interval 10

# Restart a running daemon (after config changes, if unresponsive)
scribe service restart

# Reinstall the service (use after upgrading Scribe)
scribe service reinstall
```

### Launching

```sh
scribe
```

No arguments. The TUI opens on the Dashboard view.

### Layout

```
┌────────────────────────────────────────────────────────────┐
│  Scribe  [D]ashboard [P]rojects [T]asks [O]Todos            │
│          T[R]acker   [I]nbox    [M]Reminders                │
```

The **tab bar** at the top shows all views and their shortcut keys. The

### Navigating between views

Press the indicated letter from anywhere in the TUI:

| Key | View |
|-----|------|
| `d` | Dashboard |
| `p` | Projects |
| `t` | Tasks |
| `o` | Todos |
| `r` | Tracker |
| `i` | Inbox |
| `m` | Reminders |

### Dashboard view

Shows today's tasks and the active timer. Use `j`/`k` to move the cursor,
`Space` to mark a task done.

### Projects view

Lists all active projects. Use `j`/`k` to navigate, `/` to filter by slug or
name. Press `n` to create a project (opens a form), `e` to edit, `D` to
archive (confirmation required).

### Tasks view

Lists all active, non-done tasks. Use `j`/`k` to navigate, `/` to filter by
title. Press `n` to create, `e` to edit, `D` to archive, `Space` to mark done.

### Todos view

Lists active, undone todos. Navigation is the same as Tasks.

| Key | Action |
|-----|--------|
| `n` | New todo (form) |
| `e` | Edit selected todo title |
| `D` | Archive selected todo (confirmation) |
| `Space` | Toggle done/undone |
| `v` | Move selected todo to another project (form) |

### Tracker view

Shows all non-archived time entries, most recent first.

| Key | Action |
|-----|--------|
| `Space` | Start a new timer (form) if idle; stop the running timer if active |
| `e` | Edit the note on the selected entry |
| `D` | Archive the selected entry (confirmation) |

The status bar shows the active timer slug and elapsed time, updated on each
event loop tick.

### Inbox view

Shows unprocessed capture items.

| Key | Action |
|-----|--------|
| `n` | Capture a new item (form) |
| `Enter` | Process the selected item (interactive form) |
| `D` | Delete the selected item (confirmation) |

### Reminders view

| Key | Action |
|-----|--------|
| `n` | Create a new reminder (form) |
| `e` | Edit the selected reminder |
| `D` | Archive the selected reminder (confirmation) |

### Forms

When you press `n` or `e`, a form modal overlays the current view.

- `Tab` / `Shift-Tab` — move between fields.
- Type to enter text; `Backspace` to delete.
- `Enter` — submit the form and perform the operation.
- `Esc` — cancel without making changes.

Datetime fields (e.g. in the reminder form) accept the same flexible formats as
the CLI `--at` flag. An invalid value shows an error in the status bar and
returns focus to the field.

### Confirmation dialogs

Destructive operations (`D`) open a yes/no dialog:

- `y` or `Enter` — confirm.
- `n` or `Esc` — cancel.

### Help overlay

Press `?` to open the help overlay, which lists key bindings for the current
view. Press `?` again or `Esc` to close it.

### Status bar

- Left side: active timer slug and elapsed time (e.g.
  `payments-entry-20260331-143000 — 0h 23m`), or blank when idle.
- Right side: the most recent error message in red. Press `Esc` to dismiss.

## Tab Completion

Scribe provides two layers of shell completion that work together.

### Static layer

Generated by `clap_complete`, the static layer handles:

- **Subcommand names** — `scribe <TAB>` lists `project`, `task`, `todo`, `track`, `capture`, `inbox`, `reminder`, `completions`.
- **Flag names** — `scribe task add --<TAB>` lists `--project`, `--priority`, `--due`, `--output`.
- **Enum values** — `scribe task add --priority <TAB>` lists `low  medium  high  urgent`.
  Same for `--status`, `--output`, and shell names under `scribe completions`.

### Dynamic layer

The dynamic layer is driven by a hidden `scribe __complete <entity>` subcommand
that your completion script calls at completion time. It opens the local
database and prints `<slug>\t<hint>` pairs to stdout:

| Entity argument | What is printed |
|-----------------|-----------------|
| `projects` | `slug\tproject name` for every active (non-archived) project |
| `tasks` | `slug\ttask title` for every active (non-archived) task |
| `todos` | `slug\ttitle` for every active, not-done, non-archived todo |
| `reminders` | `slug\tmessage or remind_at` for every active, non-fired reminder |
| `captures` | `slug\tbody (first 60 chars)` for every unprocessed capture item |
| `entries` | `slug\tproject_slug started_at` for every active time entry |

In **zsh** and **fish** the hint is shown as a description alongside the
candidate in the completion menu. In **bash** only the slugs are shown (no
hints; see limitation note below).

### Per-shell setup

#### zsh

```sh
# 1. Generate and install the completion script
# Use > on first install; use >! on subsequent updates (zsh noclobber)
scribe completions zsh > ~/.zfunc/_scribe

# 2. Ensure ~/.zfunc is on your fpath (add to ~/.zshrc if not already there)
echo 'fpath=(~/.zfunc $fpath)' >> ~/.zshrc
echo 'autoload -Uz compinit && compinit' >> ~/.zshrc

# 3. Reload
source ~/.zshrc
```

#### zsh with Zim

Zim's `zsh-completions` module directory is already on `fpath` — write the
file there directly, then clear the completion cache and rebuild:

```sh
scribe completions zsh >! ~/.zim/modules/zsh-completions/src/_scribe
rm -f ~/.zcompdump ~/.zcompdump.dat ~/.zcompdump.zwc
zimfw build
```

Use `>!` to force-overwrite the file — zsh's `noclobber` option (set by Zim)
blocks plain `>` if the file already exists. Deleting the `zcompdump` files
ensures zsh reloads the updated script rather than serving the old cached
version. Open a new terminal after running these commands.

No changes to `.zshrc` are needed; Zim's `completion` module calls `compinit`
automatically.

The completion script provides both static completions (subcommand names, flag
names, enum values such as `--status` and `--priority`) and dynamic completions
that query your live database for slug candidates via `scribe __complete <entity>`.

#### bash

```sh
scribe completions bash >> ~/.bash_completion
```

Or source it inline in `~/.bashrc`:

```sh
source <(scribe completions bash)
```

> **Limitation:** bash completions are static only. Slug arguments (e.g.
> `scribe task done <slug>`) will not show live candidates from the database.
> Dynamic slug completion is available in zsh and fish only.

#### fish

```sh
scribe completions fish > ~/.config/fish/completions/scribe.fish
```

The script includes `complete` directives that call `scribe __complete <entity>`
for every slug-valued argument, with `__fish_seen_subcommand_from` guards so
the correct entity is queried for each context.

### What completion looks like in zsh

```
$ scribe task done <TAB>
payments-task-fix-login        Fix the login redirect bug
payments-task-write-tests      Add unit tests for auth module
quick-capture-task-redis-rate  Look into Redis rate limiting on payments API
```

```
$ scribe task add --project <TAB>
payments       Payments Integration
api            REST API v2
quick-capture  Quick Capture
```

### Use cases

- Pipe to `jq` for custom reports.

## Sync

Scribe can mirror your full state (all projects, tasks, todos, time entries,
reminders, and capture items) to a remote provider so your data stays in sync
across multiple machines. Sync is **off by default** and requires the `sync`
Cargo feature, which is included in all pre-built binaries.

### Quick setup

Run the interactive wizard and follow the prompts:

```sh
scribe sync configure
```

Or specify the provider directly:

```sh
scribe sync configure --provider gist
```

The wizard stores all API tokens and secrets in the **OS keychain**
(macOS Keychain, Linux libsecret / gnome-keyring, Windows Credential Manager).
Nothing sensitive is written to `config.toml`.

### Supported providers

| Provider | Notes |
|---|---|
| `gist` | GitHub Gist (free, recommended — requires a GitHub PAT with `gist` scope) |
| `s3` | Any S3-compatible store: AWS S3, Cloudflare R2, MinIO, etc. |
| `icloud` | iCloud Drive file path — macOS only; iCloud handles propagation |
| `jsonbin` | JSONBin.io free-tier JSON bin |
| `dropbox` | Dropbox API v2 with a long-lived access token |
| `rest` | Self-hosted master server embedded in the Scribe daemon |
| `file` | Custom local or network file path (managed folder, NFS, Syncthing, etc.) |

Only one provider is active at a time. You can configure multiple providers and
switch between them by changing `sync.provider` in `config.toml` — credentials
for all configured providers are retained in the keychain.

### How sync works

On each sync cycle Scribe:

1. **Pulls** the remote snapshot (a JSON document of all entities).
2. **Merges** it into local state — field-level, last-write-wins on conflict,
   keyed by each entity's `updated_at` timestamp.
3. **Pushes** the merged result back as the new remote snapshot.

Entities with no `updated_at` (time entries, reminders, capture items) use
insert-or-keep: new remote entities are inserted locally; existing local
entities are never replaced.

Archiving propagates: if one machine archives a record, the `archived_at`
timestamp syncs to all other machines on the next cycle.

### Automatic sync via the daemon

If you have the background service installed, sync runs automatically:

```sh
# Install the service (runs scribe daemon in the background)
scribe service install

# Sync happens every 60 seconds by default
# Change the interval in config.toml:
# [sync]
# interval_secs = 30
```

Without the service, run sync manually at any time:

```sh
scribe sync
```

### Sync status

```sh
scribe sync status
```

Output:

```
sync enabled:  true
provider:      gist
last sync:     2026-04-02T09:01:00Z
last error:    none
```

### Removing sync configuration

```sh
# Remove stored keychain secrets for the active provider
scribe sync configure --remove
```

This clears the keychain entries. Set `sync.enabled = false` in `config.toml`
(or run `scribe sync configure` again) to reconfigure.

### Self-hosted REST master

One machine runs as the master; all others are clients. The master's HTTP
server starts automatically inside the daemon.

**On the master machine:**

```sh
scribe sync configure --provider rest
# Select [1] Master when prompted
# Enter a port (default 7171)
# A cryptographically random secret is generated and printed once:
#
#   SECRET: 4a9f2c8e1b7d3f6a...
#
# Copy this secret — it will not be shown again.
```

**On each client machine:**

```sh
scribe sync configure --provider rest
# Select [2] Client when prompted
# Enter the master URL: http://192.168.1.10:7171
# Enter the secret when prompted (input is hidden)
```

The secret is stored in each machine's keychain. Clients sync by calling
`PUT /state` (push) and `GET /state` (pull) against the master.

### GitHub Gist (recommended for most users)

1. Create a GitHub Personal Access Token at
   <https://github.com/settings/tokens> with the `gist` scope.
2. Run `scribe sync configure --provider gist` and paste the token when
   prompted.
3. On first sync, Scribe creates a **private** secret gist named
   `scribe-state.json` and stores the gist ID in `config.toml`.
4. On each subsequent machine, run `scribe sync configure --provider gist`,
   enter the same token, and add the gist ID to `[sync.gist] gist_id` in
   `config.toml` (or copy the `config.toml` from the first machine).

## Agent Integration

Scribe integrates with AI coding agents in two ways: a skill file (plain
Markdown that teaches the agent the full Scribe command surface) and an MCP
server (a real-time protocol bridge that lets agents read and write Scribe data
without spawning a subprocess for every operation).

### `scribe agent install`

Running `scribe agent install` detects which AI agent tool directories are
present on the machine and writes a `scribe.md` skill file to each one:

| Agent | Skill directory | File written |
|---|---|---|
| Claude Code / OpenCode | `~/.claude/skills/` | `scribe.md` |
| Cursor | `~/.cursor/rules/` | `scribe.md` |
| Codex | `~/.codex/` | `scribe.md` |
| Windsurf | `~/.windsurf/rules/` | `scribe.md` |

Directories that do not exist on the machine are silently skipped and reported
as `Skipped`.  Only directories that already exist receive the skill file —
`scribe agent install` never creates new directories.

After the install summary, the command prints ready-to-paste MCP configuration
snippets for Claude Code and OpenCode (see below).

### MCP server

The MCP server is included in pre-built binaries. If building from source,
add the `mcp` feature:

```sh
cargo install --path . --features mcp
```

The server uses the stdio transport.  **Do not run `scribe mcp` in a plain
terminal** — stdout is the MCP wire protocol (JSON-RPC 2.0).  Instead,
configure your agent to spawn `scribe mcp` as a child process.

#### Claude Code configuration

Append to `~/.claude/settings.json` (or create it):

```json
{
  "mcpServers": {
    "scribe": {
      "command": "scribe",
      "args": ["mcp"]
    }
  }
}
```

#### OpenCode configuration

Add to your `opencode.json`:

```json
{
  "mcp": {
    "scribe": {
      "type": "local",
      "command": ["scribe", "mcp"],
      "enabled": true
    }
  }
}
```

### MCP tools exposed

| Group | Tool | What it does |
|---|---|---|
| Projects | `project_list` | List active projects |
| | `project_create` | Create a project |
| | `project_archive` | Archive a project (cascades to items) |
| | `project_restore` | Restore an archived project |
| Tasks | `task_list` | List active tasks (optional filters) |
| | `task_create` | Create a task |
| | `task_done` | Mark a task done |
| | `task_archive` | Archive a task |
| Todos | `todo_list` | List active todos |
| | `todo_create` | Create a todo |
| | `todo_done` | Mark a todo done |
| | `todo_archive` | Archive a todo |
| Time tracking | `timer_start` | Start a timer |
| | `timer_stop` | Stop the running timer |
| | `timer_status` | Get running timer and elapsed seconds |
| | `track_report` | Generate a time report (`today`, `week`, or all-time) |
| Capture / inbox | `capture` | Quick-capture a thought |
| | `inbox_list` | List unprocessed inbox items |
| | `inbox_process` | Process an inbox item (task / todo / assign / discard) |
| Reminders | `reminder_list` | List active reminders |
| | `reminder_create` | Create a reminder |
| | `reminder_archive` | Archive a reminder |

### MCP resources exposed

| URI | Contents |
|---|---|
| `scribe://projects` | All active projects (JSON array) |
| `scribe://tasks/active` | All active non-archived tasks (JSON array) |
| `scribe://todos/active` | All active non-done todos (JSON array) |
| `scribe://timer/active` | Running timer JSON object, or `null` |
| `scribe://inbox` | Unprocessed capture items (JSON array) |
| `scribe://reminders/pending` | Active unfired reminders (JSON array) |

### A note on `scribe mcp` and stdout

The MCP wire protocol runs over stdout.  Any output that is not valid
JSON-RPC will break the connection.  Scribe prints its startup message to
**stderr** (`Scribe MCP server running on stdio. Connect your agent to this
process.`) to keep stdout clean.

## Testing and Coverage

Scribe maintains 80%+ code coverage using `cargo-llvm-cov`.

### Running Tests

```sh
# Run all tests
cargo test --all-features

# Run only unit tests
cargo test --lib

# Run only integration tests
cargo test --tests

# Run a specific integration test file
cargo test --test todo_store_tests
```

### Running with Coverage

```sh
# Install cargo-llvm-cov if not present
cargo install cargo-llvm-cov

# Generate coverage report (terminal)
cargo llvm-cov --all-features

# Generate HTML coverage report
cargo llvm-cov --all-features --html
open target/llvm-cov/html/index.html  # macOS

# Generate HTML for the whole workspace
cargo llvm-cov --all-features --workspace --html
```

### Coverage Requirements

- Minimum coverage threshold: **80%**
- Coverage is checked in CI on every PR
- Run `cargo llvm-cov --all-features --fail-under-lines 80` locally to verify before pushing

### Test Organization

Tests live in `tests/` (not in `src/`):

| Directory / Module | Purpose |
|-----------|---------|
| `scribe::testing` | Shared test helpers (temp DB, mock config) |
| `tests/*.rs` | Unit tests migrated from `src/` modules |
| `tests/tui/` | TUI integration tests (using `TestBackend`) |
| `tests/sync/` | Sync integration tests |

## Tips and workflows

### 1. Morning planning session in the TUI

```sh
scribe
```

Open the TUI, press `d` for the Dashboard, and review today's tasks. Switch to
`t` (Tasks) to scan everything outstanding. Use `/` to filter by project name.
Press `e` to bump priorities or update due dates as the day changes.

### 2. Mid-day quick capture without leaving the terminal

While in the middle of another task:

```sh
scribe capture "Look into rate limiting on the payments API"
```

One command, under a second. Return to what you were doing. Process the inbox
at the end of the day.

### 3. End-of-day time tracking report

```sh
# See all time logged today
scribe track report --today

# Narrow to one project
scribe track report --today --project payments

# Machine-readable for further processing
scribe track report --week --output json
```

### 4. Archiving a completed project

```sh
# 1. Mark the project complete
scribe project edit payments --status completed

# 2. Archive it (cascades to all linked items)
scribe project archive payments

# 3. Verify
scribe project list --archived
```

### 5. Using `--output json` with `jq` for custom reports

List all high-priority tasks across all projects, showing only slug and title:

```sh
scribe task list --priority high --output json \
  | jq -r '.[] | "\(.slug)\t\(.title)"'
```

Count tasks per status:

```sh
scribe task list --output json \
  | jq 'group_by(.status) | map({status: .[0].status, count: length})'
```

Find tasks due this week:

```sh
scribe task list --output json \
  | jq --arg today "$(date +%Y-%m-%d)" \
       '[.[] | select(.due_date != null and .due_date <= $today)]'
```
